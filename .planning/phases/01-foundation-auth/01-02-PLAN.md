<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Unvalidated registration/login credentials enter the system |
| API→auth service | Password handling and token generation/verification |
| auth service→database | User credential storage and retrieval |
| client→protected endpoints | Access to protected resources after authentication |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | S (Spoofing) | Authentication endpoints | mitigate | Implement JWT verification, strong password hashing, and rate limiting |
| T-02-02 | T (Tampering) | JWT tokens | mitigate | Use strong secret, proper expiration, and signature verification |
| T-02-03 | I (Information Disclosure) | Password storage | mitigate | Use bcrypt with appropriate salt rounds, never store plaintext passwords |
| T-02-04 | D (Denial of Service) | Auth endpoints | mitigate | Implement rate limiting on auth endpoints (can be added in middleware) |
| T-02-05 | E (Elevation of Privilege) | Role-based access | mitigate | Verify role on every protected endpoint using middleware |
| T-02-06 | R (Repudiation) | Auth logs | accept | Basic logging sufficient for MVP; detailed audit logging in future |
</threat_model>
